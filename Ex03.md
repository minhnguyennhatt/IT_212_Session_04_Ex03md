## 1. Phân tích lý do prompt thô dễ bỏ sót lỗi

Prompt thô:

```text
Mã này bị lỗi gì?
```

Prompt này quá chung chung nên dễ khiến AI bỏ sót lỗi logic vì các lý do sau:

Thứ nhất, đoạn mã Java này **không sai cú pháp**. Code vẫn có thể compile bình thường, không báo lỗi biên dịch. Vì vậy, nếu chỉ hỏi chung chung “mã này bị lỗi gì?”, AI có thể chỉ kiểm tra bề mặt như cú pháp, kiểu dữ liệu, tên hàm, vòng lặp, mà không đi sâu vào hành vi thực tế khi chạy.

Thứ hai, lỗi trong đoạn code là **lỗi logic**, không phải lỗi runtime rõ ràng như `NullPointerException` hay `ArrayIndexOutOfBoundsException`. Cụ thể, vòng lặp bên trong bắt đầu từ:

```java
int j = i;
```

Điều này khiến phần tử luôn tự so sánh với chính nó:

```java
arr[i] == arr[i]
```

Biểu thức này luôn đúng, nên hàm sẽ trả về `arr[0]` ngay từ lần lặp đầu tiên.

Thứ ba, prompt thô **không cung cấp test case cụ thể**. Nếu không có ví dụ như mảng `{1, 2, 3, 4}` không có phần tử trùng nhưng hàm vẫn trả về `1`, AI có thể không nhận ra hành vi sai của chương trình.

Thứ tư, prompt thô **không nêu rõ kỳ vọng nghiệp vụ**. Hàm cần tìm phần tử trùng lặp đầu tiên, nhưng prompt không yêu cầu AI kiểm tra xem logic hiện tại có thực sự đáp ứng mục tiêu đó không.

Thứ năm, prompt thô **không yêu cầu tối ưu hiệu năng**. Nếu AI có phát hiện lỗi, có thể chỉ sửa `j = i + 1`, nhưng vẫn giữ thuật toán hai vòng lặp với độ phức tạp `O(N²)`, trong khi đề bài yêu cầu cải thiện bằng `HashSet` xuống `O(N)`.

## 2. Nội dung Prompt tối ưu mới thiết kế

````markdown
Bạn hãy đóng vai trò là một Code Auditor chuyên phân tích lỗi logic trong mã Java.

Bối cảnh:
Tôi có một hàm Java dùng để tìm phần tử trùng lặp đầu tiên trong một mảng số nguyên. Tuy nhiên, hàm hiện tại có thể chạy đúng cú pháp và không báo lỗi compile, nhưng kết quả trả về lại sai về mặt logic.

Mã nguồn hiện tại:

```java
public class DuplicateFinder {

    // Lỗi logic: j bắt đầu từ i dẫn đến việc phần tử tự so sánh với chính nó
    public static Integer findDuplicate(int[] arr) {
        for (int i = 0; i < arr.length; i++) {
            for (int j = i; j < arr.length; j++) {
                if (arr[i] == arr[j]) {
                    return arr[i];
                }
            }
        }
        return null;
    }
}
```
````

Ca kiểm thử cụ thể:
Với mảng đầu vào:

```java
int[] arr = {1, 2, 3, 4};
```

Mảng này không có phần tử trùng lặp, kết quả đúng phải là `null`. Tuy nhiên, code hiện tại lại trả về `1`.

Yêu cầu:

1. Phân tích nguyên nhân gốc rễ của lỗi logic.
2. Giải thích vì sao vòng lặp hiện tại khiến phần tử tự so sánh với chính nó.
3. Chỉ ra vì sao prompt chung chung như “Mã này bị lỗi gì?” dễ khiến AI bỏ sót lỗi này.

5. Tối ưu độ phức tạp từ `O(N²)` xuống `O(N)`.
6. Giữ nguyên mục tiêu của hàm: trả về phần tử trùng lặp đầu tiên nếu có, nếu không có thì trả về `null`.

## 3. Mã nguồn Java đã sửa hoàn chỉnh sử dụng HashSet

```java
import java.util.HashSet;
import java.util.Set;

public class DuplicateFinder {

    public static Integer findDuplicate(int[] arr) {
        if (arr == null || arr.length == 0) {
            return null;
        }

        Set<Integer> seenNumbers = new HashSet<>();

        for (int number : arr) {
            if (seenNumbers.contains(number)) {
                return number;
            }

            seenNumbers.add(number);
        }

        return null;
    }
}
````

## 4. Giải thích mã nguồn sau khi sửa

Trong phiên bản ban đầu, lỗi nằm ở vòng lặp bên trong:

```java
for (int j = i; j < arr.length; j++)
```

Khi `j = i`, chương trình sẽ so sánh:

```java
arr[i] == arr[j]
```

Tức là:

```java
arr[i] == arr[i]
```

Điều kiện này luôn đúng, nên ngay tại `i = 0`, `j = 0`, hàm trả về phần tử đầu tiên của mảng.

Ví dụ với mảng:

```java
int[] arr = {1, 2, 3, 4};
```

Lần lặp đầu tiên:

```java
i = 0
j = 0
arr[0] == arr[0] // 1 == 1, luôn đúng
```

Vì vậy hàm trả về `1`, dù mảng không có phần tử trùng lặp nào.

## 5. Vì sao dùng HashSet tốt hơn?

Phiên bản đã sửa sử dụng `HashSet` để lưu các phần tử đã duyệt qua.

Ý tưởng xử lý:

- Duyệt từng phần tử trong mảng.
- Nếu phần tử đã tồn tại trong `HashSet`, nghĩa là phần tử đó bị trùng.
- Trả về phần tử trùng đầu tiên được phát hiện.
- Nếu duyệt hết mảng mà không có phần tử nào trùng, trả về `null`.

Ví dụ:

```java
int[] arr = {1, 2, 3, 4};
```

Kết quả:

```java
null
```

Ví dụ khác:

```java
int[] arr = {1, 2, 3, 2, 4};
```

Kết quả:

```java
2
```

## 6. Độ phức tạp trước và sau khi sửa

| Phiên bản        | Cách xử lý                | Độ phức tạp thời gian |
| ---------------- | ------------------------- | --------------------- |
| Code lỗi ban đầu | Dùng 2 vòng lặp lồng nhau | `O(N²)`               |
| Code đã sửa      | Dùng `HashSet`            | `O(N)`                |

Phiên bản mới nhanh hơn vì chỉ cần duyệt mảng một lần. Tuy nhiên, nó sử dụng thêm bộ nhớ để lưu các phần tử đã gặp, nên độ phức tạp bộ nhớ là `O(N)`.
